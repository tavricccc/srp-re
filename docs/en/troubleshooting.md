# Troubleshooting

[繁體中文](../troubleshooting.md) · [Documentation home](../README.md)

Capture the first failure, request ID, environment, commit SHA, and time. Later errors are often consequences. Never post tokens or personal data publicly.

## Installation and checks

- **`npm ci` fails:** use Node.js 24, keep the lockfile generated by the expected npm version, and inspect registry/proxy settings. Do not hand-edit the lockfile.
- **Generated files differ:** run `npm run generate:all`, inspect source and output, and commit both. CI requires a clean diff after generation.
- **Deno is missing:** run `npm ci`, then `npm run check:edge` so the project-pinned `deno-bin` is used.

## Sign-in

| Symptom | Check |
| --- | --- |
| Google popup rejected | Google provider, authorized domain, browser popup policy |
| Domain rejected | Matching `VITE_ALLOWED_DOMAIN` and `ALLOWED_DOMAIN` |
| Sign-in succeeds, sync fails | Firebase project, Web API key, service account, verified email |
| Admin remains a user | `ADMIN_EMAILS`, allowed domain, refreshed sign-in/token |

Do not hard-code a client role or disable backend checks.

## API and data

For universal 401/403 responses, verify a current ID token from the configured Firebase project, clock synchronization, and matching backend credentials. For missing records, check category access, review state, authorship, and administrator role. Distinguish RLS client results from service-role queries.

If Realtime stops updating, verify the initial read, CSP/proxy connectivity, and event audience. Selecting the active desktop sidebar item or mobile bottom navigation item forces a list refresh. A refresh that repairs the page usually points to subscription or patch behavior rather than lost data.

## Images

1. Check source size, pixels, and count against `rate-limits.config.json`.
2. Confirm the upload-session action succeeded.
3. Confirm an authenticated Cloudinary resource exists.
4. Check webhook URL, secret, and Function logs.
5. If ready but invisible, inspect URL resolution, expiry, and CSP.

Never make delivery permanently public or expose the API secret to fix rendering.

## Notifications and workers

If in-app notifications work but Push fails, inspect browser permission, service worker, VAPID, FCM service account, and token/topic state. For a growing outbox, inspect worker triggers, `WEBHOOK_SECRET`, Upstash, and vendor status. For Notion, confirm the integration still has database access and expected properties. Confirm handler idempotency before retries.

## Deployment stages

| Stage | Common cause |
| --- | --- |
| Secret validation | Wrong Environment, name, or empty value |
| Supabase link/push | Token, ref, password, or migration history |
| Function deploy | Deno type/import failure or missing secret |
| Healthcheck | Mismatched `WEBHOOK_SECRET` or deployment not ready |
| Vercel build | Token scope, org/project ID, or public env values |
| Backend wait timeout | Matching backend workflow failed or did not trigger |

If unresolved, open a minimal issue following the [contributing guide](contributing.md), including OS, Node version, commit, command, redacted first error, and expected/actual behavior. Report vulnerabilities [privately](security.md#reporting-a-vulnerability).
